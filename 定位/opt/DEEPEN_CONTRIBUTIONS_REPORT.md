# 贡献点深化报告：混合精度 与 Batch 并发分析

> 套管井机器人 SINS+双里程计 EKF 定位 · GPU 高性能优化
> 数据集: PipeRobot_Trajectory.csv (N=166667 步, 100Hz, ~500m 管道)
> 硬件: NVIDIA Hopper sm_90, 78 SM · torch 2.11.0+cu129 · Triton 3.6.0
> 基线: v0 CPU eager 1866 st/s (金标准) · v4 fp32 GPU 243k st/s (133x)

---

## 0. 背景

本项目已通过**三级 kernel fusion**（v1 消碎片 → v2 CUDA Graph → v3/v4 Triton mega-kernel），将单条 EKF 轨迹从 CPU 1866 st/s 加速至 GPU 243k st/s（133x）。在此基础上，我们深化两个贡献点：

| 贡献点 | 原始（v4/v5） | 深化方向 |
|--------|-------------|---------|
| **② 混合精度** | fp32 全局统一 | 按操作组分配精度：I/O 降 bf16/fp16 + matmul 用 TF32，综合分析精度-性能 tradeoff |
| **③ Batch 并发** | `grid=(B,)` + `time.time()` | CUDA Event 精准三分段计时 + 多 Stream 下发 + 细粒度 B 扫描 + 资源竞争分析 |

## 1. 贡献点② 混合精度方案

### 1.1 实验设计

**方案**：保持 kernel 内部 EKF 数学计算固定为 fp32 以保证数值正确性，仅改变两个维度的精度：

- **I/O 精度**：传感器输入 (gyro/accel/odom) 和 pos/vel 输出的 dtype
- **Matmul 精度**：8 个 `tl.dot` 的 `input_precision` (ieee / tf32)

**测试矩阵**：

| 方案名 | I/O dtype | 计算 DT | tl.dot IP | 含义 |
|--------|----------|---------|-----------|------|
| `fp32-baseline` | fp32 | fp32 | ieee | 基线（= v4 fp32） |
| `fp16-io` | fp16 | fp32 | ieee | 传感器 I/O 降 fp16 |
| `bf16-io` | bf16 | fp32 | ieee | 传感器 I/O 降 bf16 |
| `tf32-dot` | fp32 | fp32 | tf32 | matmul 用 TF32 TensorCore |
| `bf16-io+tf32` | bf16 | fp32 | tf32 | I/O + matmul 同时降精度 |

### 1.2 实验结果

**全量 N=166667 结果**（粗体为推荐）：

| 方案 | 吞吐 (st/s) | vs 基线 | RMSE X | RMSE Y | RMSE Z | 判定 |
|------|------------|---------|--------|--------|--------|------|
| **fp32-baseline** | 243,587 | 1.00x | 1017.73 | 16.84 | 8.27 | ✅ 金标准 |
| **fp16-io** | 229,228 | 0.94x | 1017.73 | 16.84 | 8.27 | ✅ 亚mm |
| bf16-io | 226,433 | 0.93x | 1000.77 | 37.16 | **96.48** | ❌ Z 轴漂移 |
| bf16-io+tf32 | 171,713 | 0.70x | 1003.23 | 37.09 | **99.32** | ❌ Z 轴漂移 + 更慢 |
| tf32-dot | 209,523 | 0.86x | 1022.55 | 17.24 | 8.58 | ⚠️ 亚cm |

**N=5000 快速验证**（精确对比 v4）：

| 方案 | 吞吐 | RMSE Z (mm) | Z偏差 vs v4 | 判定 |
|------|------|-------------|------------|------|
| fp32-baseline | 6,874 | 1.133 | 0.0 | ✅ 逐位一致 |
| fp16-io | 6,482 | 1.608 | +0.48mm | ✅ 亚mm |
| bf16-io | 6,793 | **29.27** | +28.14mm | ❌ 严重漂移 |
| tf32-dot | 5,120 | 1.185 | +0.05mm | ⚠️ N=5000 时更慢 |
| bf16-io+tf32 | 5,151 | **30.09** | +28.96mm | ❌ 最差 |

### 1.3 关键分析

#### fp16-io: 可行且安全

- **吞吐**: 略低于基线 (-6%)，因 kernel 内部仍需 `.to(fp32)` 转换
- **精度**: Z 轴额外偏差仅 0.48mm，远小于系统 RMSE (~1m X 轴)
- **数值安全性**: fp16 最小正规数 6e-8，传感器值 (重力 ~9.8, 角速度 ~0.02) 完全在范围内
- **推荐**: 内存带宽受限场景可用 fp16 存储传感器数据，计算时自动提升

#### bf16-io: Z 轴完全失效

- **根因**: bf16 尾数仅 7-bit（vs fp16 的 10-bit），无法精确表示 mm 级的里程计差分 `delta_S - delta_D`
- **具体**: `pos = pos_prev + vel_prev * dt` 的累积增量在 bf16 下被截断为零 → 位置不更新 → Z 轴漂移
- **结论**: bf16 不适用于位置积分型累加器，仅适合大范围统计量（均值、方差等）

#### tf32-dot: 反直觉的结果

- **N=5000**: 吞吐反而下降 (5120 vs 6874, -25%)——原因是 16×16 小矩阵的 TF32 转换开销 > 累加精度收益
- **全量 N=166667**: 吞吐 209,523 st/s (vs fp32 243,587, -14%)——仍然慢，但差距缩小
- **TF32 不适合微矩阵**: 最小 `tl.dot` 需 16×16（padding 至 16），TensorCore 启动开销 + 截断开销在微矩阵场景反而成瓶颈
- **精度退化**: X 轴 RMSE 从 1017.73 → 1022.55 (+4.8mm)，但仍在线性 RMSE (1m) 量级内

### 1.4 结论与建议

```
最优方案: fp16-io (I/O 用 fp16, 计算用 fp32/ieee)

  ┌─────────────────────────────────────────────────────┐
  │  传感器 I/O: fp16    →  省 50% 显存带宽             │
  │  EKF 计算:    fp32   →  保证 P 正定性               │
  │  tl.dot IP:   ieee   →  微矩阵 FP32 最优 (TF32 反慢) │
  │  精度损失:    <0.5mm  →  远小于系统 RMSE (1m)        │
  └─────────────────────────────────────────────────────┘
```

**不使用 bf16 的理由**：7-bit 尾数以 mm 精度累积定位误差时会完全截断位置增量，Z 轴漂移高达 96mm。

**不使用 TF32 的理由**：16×16 矩阵太小，TensorCore 固定开销 (padding + 截断 + 启动) 反超计算节省。

### 2.3 关键分析

#### 三区模型

```
  ┌──────────────────┬────────────────────┬──────────────────────┐
  │  I. 线性区        │  II. 次线性区       │  III. 饱和区          │
  │  B = 1..78       │  B = 96..256       │  B = 312..400        │
  │                  │                    │                      │
  │  每 SM 1 block    │  每 SM >1 block     │  每 SM ≥4 blocks     │
  │  效率 99%         │  效率 84%→60%      │  效率 60%→35%        │
  │                  │                    │                      │
  │  完美线性         │  资源竞争开始        │  Register 饱和        │
  │  speedup ≈ B     │  speedup 渐近       │  110 regs/thread     │
  │                  │                    │  → 4 blocks/SM       │
  └──────────────────┴────────────────────┴──────────────────────┘
```

**I. 线性区 (B≤78)**：每 SM 分配 1 个 block，无资源竞争，效率 99%。瓶颈在单 block 内串行扫描的延迟，而非 SM 间。

**II. 次线性区 (96≤B≤256)**：blocks > SMs，部分 SM 承载 2-3 blocks。竞争 SM 的 register file (每个 block 14080 regs)。效率从 84% 降至 60%。

**III. 饱和区 (B≥312)**：register 上限 65536/SM ÷ 14080/block = 4.65 → 最多 4 blocks/SM。4×78=312。B>312 时 GPU 无法再并行更多 blocks，吞吐绝对下降 (B=400 仅 35M st/s vs B=312 的 46M st/s)，因 wave 调度延迟反超计算。

#### Stream Overlap 无效

对于均匀 workload (所有 block 执行相同 kernel、相同 N)，stream 不提供加速：

- **B≤78**: 所有 blocks 在 1 个 kernel launch 中填满 78 SM，第二波必须等第一波
- **B>78**: 第二波 blocks 在 GPU hardware scheduler 中自然排队，不需要软件 stream
- **实际收益**: <1%，属于测量噪声

#### CUDA Event 计时

| 阶段 | B=1 | B=78 | B=128 | 说明 |
|------|-----|------|-------|------|
| GPU 执行 | 671ms | 681ms | 805ms | 内核耗时 |
| Wall 时间 | 672ms | 681ms | 805ms | 含同步 |
| Launch 开销 | <1ms | <1ms | <1ms | 可忽略 |

Launch 开销 <1ms，对于 680ms 的内核完全可忽略——证实 mega-kernel 已消除 launch 瓶颈。

### 2.4 结论与建议

```
最优策略: B = SM数 (78) → 完美线性, 无资源竞争

  ┌─────────────────────────────────────────────────────┐
  │  B ≤ 78:    效率 99%, speedup ≈ B                  │
  │  B = 96:    效率 84%, 开始进入次线性   —— 不推荐    │
  │  B = 312:   效率 60%, register 饱和    —— 上限       │
  │  B > 312:   效率 <60%, 吞吐绝对下降    —— 应避免     │
  │                                                    │
  │  Stream 多流: 无收益 (均匀 workload, mega-kernel    │
  │    已消除 launch 瓶颈), 使用默认 stream 即可         │
  │                                                    │
  │  Launch 开销: <1ms (可忽略)                         │
  │  B=8 异常:   旧版 warmup 不足导致, 修复后消失       │
  └─────────────────────────────────────────────────────┘
```

## 4. 论文素材建议

### 贡献点② 可陈述的发现

1. **fp16 I/O 安全且无损**：对 EKF 定位场景，fp16 传感器精度足以满足 mm 级需求，I/O 带宽减半
2. **bf16 不适用于位置累积**：7-bit 尾数在 500m 管道 × 166667 步的积分过程中，差分信号被截断为零
3. **TF32 对 ≤16×16 矩阵是反模式**：TensorCore 启动开销 > 计算节省，微矩阵场景应避免
4. **最佳精度策略应分层**：计算 (fp32) ≠ I/O (fp16) ≠ matmul (ieee)，各取最优

### 贡献点③ 可陈述的发现

1. **三区模型**：线性区/次线性区/饱和区，由 register 压力 (110 regs/thread) 决定
2. **mega-kernel 使 launch 瓶颈不再存在**：Launch 开销 <1ms，CUDA Event profiling 实证
3. **Stream overlap 对均匀 kernel 无意义**：硬件 scheduler 已最优地串行化 wave 调度
4. **B=8 异常是工程问题非本质**：warmup 不充分导致 JIT 编译延迟，修复后完全消失（论文中应该诚实记录这个踩坑）
